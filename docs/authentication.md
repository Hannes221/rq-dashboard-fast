# Authentication & Authorization

RQ Dashboard FastAPI supports opt-in, token-based authentication with per-queue access control. When no auth config is provided, the dashboard runs with open access (the default behavior).

## Quick Start

### 1. Generate a token

```bash
rq-dashboard-fast generate-token
```

Output:

```
Token (give to the user):  3f0d1c4b-a8e7-4f2d-9b6a-1234abcd5678
Hash  (put in config):     f4d8e72a9c...
```

### 2. Create a config file

Create an `auth.yaml` file:

```yaml
tokens:
  - hash: "f4d8e72a9c..."   # paste the hash from step 1
    queues: ["*"]
    access: admin
```

### 3. Start the dashboard with auth enabled

```bash
rq-dashboard-fast --auth-config auth.yaml
```

Or via environment variable:

```bash
export RQ_DASH_AUTH_CONFIG=auth.yaml
rq-dashboard-fast
```

Or in Python:

```python
from fastapi import FastAPI
from rq_dashboard_fast import RedisQueueDashboard

app = FastAPI()
dashboard = RedisQueueDashboard(
    redis_url="redis://localhost:6379",
    prefix="/rq",
    auth_config="auth.yaml",
)
app.mount("/rq", dashboard)
```

### 4. Access the dashboard

Open `http://localhost:8000/rq?token=3f0d1c4b-a8e7-4f2d-9b6a-1234abcd5678` in your browser. The token is exchanged for a session cookie and stripped from the URL immediately.

## Config File Reference

```yaml
tokens:
  - hash: "f4d8e72..."          # SHA-256 hash of the token (required)
    queues: ["emails", "notifications"]  # queues this token can see (required)
    access: read                 # "read" or "admin" (default: "read")
    title: "Email Team Dashboard"  # custom page title (optional)

  - hash: "a1b2c3d..."
    queues: ["*"]                # wildcard — access to all queues
    access: admin
```

### Fields

| Field | Required | Description |
|-------|----------|-------------|
| `hash` | Yes | SHA-256 hex digest of the token UUID. Generated by `rq-dashboard-fast generate-token`. |
| `queues` | Yes | List of queue names this token can access. Use `["*"]` for all queues. |
| `access` | No | `read` (view only) or `admin` (view + delete, requeue, clear). Defaults to `read`. |
| `title` | No | Custom title shown in the header and the browser tab. |

### Access Levels

- **`read`** — View queues, jobs, and workers. Export data. Delete/requeue/clear buttons are hidden.
- **`admin`** — Everything in `read`, plus delete jobs, requeue failed jobs, and clear queues.

## How Authentication Works

1. **Token in URL** — User visits `/rq?token=<uuid>`. The middleware hashes the token with SHA-256, looks it up in the config, sets an `HttpOnly` session cookie (`SameSite=Lax`), and redirects to the same page without the `?token=` parameter. This keeps the raw token out of browser history, server logs, and referrer headers.

2. **Session cookie** — Subsequent requests authenticate via the cookie. No token in the URL is needed after the initial visit.

3. **Login page** — If auth is enabled and no valid token or cookie is present, the user is redirected to a simple "Enter your token" page.

4. **No config** — If no `auth_config` path is provided (or the file doesn't exist), authentication is completely disabled and the dashboard works as it always has.

## CSRF Protection

Mutation requests (delete job, requeue job, clear queue) are protected against Cross-Site Request Forgery:

- A CSRF token is generated per session and stored in an `HttpOnly` cookie.
- The token is embedded in every page via a `<meta name="csrf-token">` tag.
- The dashboard's JavaScript includes the token as an `X-CSRF-Token` header on all mutation requests.
- The server rejects mutation requests where the header doesn't match the cookie.

This is handled automatically — no action needed from users or integrators.

## Examples

### Team with scoped read-only access

Give the email team a token that only shows their queues:

```bash
rq-dashboard-fast generate-token
# Token: aaaa-bbbb-cccc-dddd
# Hash:  1111abcd...
```

```yaml
tokens:
  - hash: "1111abcd..."
    queues: ["emails", "notifications"]
    access: read
    title: "Email Team Dashboard"
```

The email team sees only the `emails` and `notifications` queues. Delete and requeue buttons are hidden. The page header and browser tab both show "Email Team Dashboard".

### Admin with full access

```bash
rq-dashboard-fast generate-token
# Token: eeee-ffff-gggg-hhhh
# Hash:  2222efgh...
```

```yaml
tokens:
  - hash: "1111abcd..."
    queues: ["emails", "notifications"]
    access: read
    title: "Email Team Dashboard"

  - hash: "2222efgh..."
    queues: ["*"]
    access: admin
```

The admin token sees all queues with full delete/requeue/clear capabilities.

### TV/kiosk dashboard via bookmark

Bookmark or iframe the URL with the token:

```
http://dashboard.internal:8000/rq?token=aaaa-bbbb-cccc-dddd
```

On first load, the token is exchanged for a cookie and stripped from the URL. The dashboard continues to work via the cookie for the session lifetime.

## Docker

Pass the config file as a volume mount:

```bash
docker run \
  -e REDIS_URL=redis://redis:6379 \
  -e RQ_DASH_AUTH_CONFIG=/etc/rq-dashboard/auth.yaml \
  -v ./auth.yaml:/etc/rq-dashboard/auth.yaml:ro \
  hannes221/rq-dashboard-fast
```

Or in Docker Compose:

```yaml
services:
  dashboard:
    image: hannes221/rq-dashboard-fast
    ports:
      - "8000:8000"
    environment:
      - REDIS_URL=redis://redis:6379
      - RQ_DASH_AUTH_CONFIG=/etc/rq-dashboard/auth.yaml
    volumes:
      - ./auth.yaml:/etc/rq-dashboard/auth.yaml:ro
```

## Security Notes

- Tokens are never stored on the server — only their SHA-256 hashes are in the config file.
- The raw token appears in the URL only briefly during the initial redirect, then is replaced by a session cookie.
- Cookies are set with `HttpOnly` (no JavaScript access), `SameSite=Lax`, and `Secure` when served over HTTPS.
- CSRF protection is enforced on all state-changing operations.
- For production deployments, serve the dashboard behind HTTPS (e.g., via a reverse proxy) so that cookies and tokens are encrypted in transit.
